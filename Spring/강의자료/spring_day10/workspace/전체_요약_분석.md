# Spring Day10 ThreeTier 프로젝트 전체 요약

## 1) 분석 기준과 범위
- 기준 프로젝트: `threetier_v4_수정_삭제_조회_카카오로그아웃`
- 비교 대상: `threetier_v1_페이징`, `threetier_v2_더보기_무한스크롤`, `threetier_v3_필터`, `threetier_v4_수정_삭제_조회_카카오로그아웃`

## 2) 전체 구조 요약 (v4 기준)
- 아키텍처: `Controller -> Service -> DAO -> Mapper(XML) -> MySQL`
- 화면 구성: 서버 렌더링(Thymeleaf) + JS 비동기 fetch 혼합
- 핵심 연결:
  - 목록 API: `PostAPIController` -> `PostService.list()` -> `postMapper.xml selectAll/selectTotal`
  - 작성/수정/삭제: `PostController` -> `PostService` -> `PostDAO/TagDAO/FileDAO/PostFileDAO`
  - 회원/로그인: `MemberController` -> `MemberService` -> `MemberDAO` -> `memberMapper.xml`
  - 카카오 OAuth: `member/login` 페이지 JS -> 카카오 인증 -> `/kakao/login` -> `KaKaoService`

## 3) 데이터 모델 연결고리
- `tbl_member`: 회원 기본 정보
- `tbl_oauth`: OAuth 제공자 정보 (회원 1:1, `id` 공유)
- `tbl_post`: 게시글 (`member_id` FK -> `tbl_member.id`)
- `tbl_tag`: 태그 (`post_id` FK -> `tbl_post.id`)
- `tbl_file`: 첨부파일 메타데이터
- `tbl_post_file`: 게시글-파일 매핑 (`id`는 `tbl_file.id`, `post_id`는 `tbl_post.id`)
- `view_post_file`: 파일 + 게시글 매핑 조회용 뷰

## 4) 기능별 데이터 흐름

### 4.1 게시글 작성
1. `post/write` 폼 전송
2. `PostController.write()` -> `PostService.write()`
3. `tbl_post` insert (생성 키 획득)
4. 태그 리스트를 `tbl_tag`에 insert
5. 파일마다 `tbl_file` insert 후 `tbl_post_file` insert
6. 실제 파일을 `C:/file/yyyy/MM/dd` 경로에 저장

### 4.2 게시글 목록 + 검색 + 태그 필터 + 무한스크롤
1. 브라우저 JS가 `/api/posts/list/{page}?type=&keyword=&tagNames=` 호출
2. `PostAPIController.list()`가 `Search` 객체 바인딩
3. `PostService.list()`에서 `Criteria` 생성, 총개수/목록 조회
4. `postMapper.xml` 동적 SQL로 키워드/타입/태그 필터 적용
5. 각 게시글에 태그/파일 목록을 추가 조립 후 JSON 반환
6. 프론트는 페이지 1이면 치환 렌더링, 이후 스크롤은 누적 렌더링

### 4.3 게시글 상세/수정/삭제
- 상세: `PostService.detail()`에서 본문 + 태그 + 파일 조회
- 수정:
  - 본문 update
  - 신규 태그/파일 추가 저장
  - 프론트에서 전달한 `tagIdsToDelete`, `fileIdsToDelete` 삭제 처리
  - 파일 삭제는 물리 파일 삭제 + `tbl_post_file` + `tbl_file` 삭제
- 삭제:
  - 태그 전체 삭제
  - 연결 파일 물리 삭제 + 매핑/파일 테이블 삭제
  - 게시글 삭제

### 4.4 회원가입/로그인/로그아웃
- 일반 회원가입: `tbl_member` insert -> `tbl_oauth(provider=threetier)` insert
- 카카오 회원가입: `provider=kakao`로 저장
- 로그인: 이메일/비밀번호로 `tbl_member + tbl_oauth` 조인 조회
- remember 체크 시 쿠키(`remember`, `remember-member-email`) 저장
- 로그아웃: 세션 무효화

### 4.5 카카오 로그인
1. 로그인 페이지 JS가 카카오 authorize URL로 이동
2. 카카오가 `/kakao/login?code=...`로 리다이렉트
3. `KaKaoService`가 토큰 요청 -> 사용자 정보 요청
4. 이메일 기준 기존 회원 조회
  - 없으면: 카카오 추가정보 입력 페이지 이동
  - 있으면: 세션 로그인 처리 후 게시글 목록 이동

## 5) 버전별 비교표

| 버전 | 핵심 기능 | 주요 엔드포인트 | DB/쿼리 변화 | 프론트 변화 |
|---|---|---|---|---|
| `v1_페이징` | 기본 회원/게시글 흐름 + 페이지네이션 | `/post/list`(페이지 기반), `/post/write`, `/member/login` | 게시글/태그/파일 기본 테이블 사용, 페이징용 조회 | 리스트 렌더링 + 기본 페이지 이동 |
| `v2_더보기_무한스크롤` | 더보기/무한스크롤 도입 | `/api/posts/list/{page}` 비동기 조회 강화 | `Criteria`에서 `count=rowCount+1` 패턴으로 `hasMore` 계산 | `post/list/event.js`에서 스크롤 감지 후 추가 fetch |
| `v3_필터` | 검색 + 태그 필터 | `/api/posts/list/{page}?type=&keyword=&tagNames=` | `postMapper.xml` 동적 SQL(`search.keyword`, `search.tagNames`) 추가 | 체크박스 태그 다중선택, 검색 버튼 연동 |
| `v4_수정_삭제_조회_카카오로그아웃` | 상세/수정/삭제 + 카카오 로그인/로그아웃 | `/post/detail`, `/post/update`, `/post/delete`, `/kakao/login`, `/member/logout` | 수정 시 `tagIdsToDelete`, `fileIdsToDelete` 처리, 파일 실제 삭제 + DB 삭제 | 수정 화면에서 태그/이미지 클릭 삭제 수집 후 hidden input 전송 |

## 6) 버전 진행 방향 요약
- `v1 -> v2`: 정적 페이징 중심에서 동적 로딩 UX로 확장
- `v2 -> v3`: 단순 목록에서 복합 필터 검색으로 확장
- `v3 -> v4`: 조회 중심에서 수정/삭제/상세 + OAuth 로그인까지 확장

## 7) 수업(실습) 포인트

### 7.1 v1 실습 포인트
- 3계층 분리 기본기
- `Criteria` 페이징 계산 로직
- 서버 렌더링 기반 게시글 목록/작성 흐름

### 7.2 v2 실습 포인트
- 서버 페이징 + 클라이언트 누적 렌더링 역할 분리
- `hasMore` 계산 패턴(`rowCount+1`)
- `fetch` 비동기 이벤트 처리

### 7.3 v3 실습 포인트
- `Search` DTO로 복합 조건 바인딩
- MyBatis 동적 SQL(`<if>`, `<choose>`, `<foreach>`)
- 태그 다중선택과 쿼리스트링 직렬화

### 7.4 v4 실습 포인트
- 상세/수정/삭제까지 CRUD 흐름 완성
- 수정 시 추가/삭제 동시 처리(`tagIdsToDelete`, `fileIdsToDelete`)
- 물리 파일 처리와 DB 트랜잭션 연결
- OAuth 콜백 처리 + 세션 연동

## 8) 현재 코드 기준 리스크 및 개선 포인트
1. 총개수 쿼리 정확도 이슈 가능
- `selectTotal`이 `tbl_post join tbl_tag` 기반이라 태그 없는 글 누락/중복 카운트 가능성 존재

2. 목록 조회 N+1 가능성
- 목록 루프마다 태그/파일 추가 조회 수행

3. 파일 경로 보안 취약 가능성
- `/api/files/display`에서 입력 파라미터를 경로로 직접 사용

4. 카카오 실패 시 NPE 위험
- `KaKaoService.getKakaoInfo()`가 `null` 반환 가능, 컨트롤러에서 즉시 접근

5. 권한 검증 강화 필요
- 수정/삭제 시 작성자 검증 로직 보완 필요

6. 인증 보안 개선 필요
- 비밀번호 평문 비교 방식 개선(해시) 필요

## 9) 결론
- 이 프로젝트는 버전이 올라가며 `기본 CRUD -> 비동기 UX -> 필터링 -> 수정/삭제/상세 + OAuth`로 단계적으로 확장된 학습형 3계층 애플리케이션이다.
- 특히 v4는 기능 완성도가 높지만, 실제 서비스 수준으로 가려면 보안/권한/쿼리 최적화 보강이 필요하다.
